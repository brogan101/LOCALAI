# JARVIS_DECISIONS

Purpose: record durable architecture decisions so future prompts and local AI agents do not re-litigate or forget them. Keep entries short.

| Date | Decision | Reason | Impact |
|---|---|---|---|
| 2026-04-25 | Build on the existing `brogan101/LOCALAI` repo, not a new app. | Phase 00 verified an existing Express API, React/Vite control center, SQLite schema, model routing, RAG, voice, web, integrations, updater/repair/observability/tasks/rollback/plugins/WorldGUI surfaces. | All phases must extend existing systems first and avoid replacement scaffolds. |
| 2026-04-25 | Local-first by default; API keys optional only. | User wants no required cloud cost and local data control. | Cloud/API providers require data classification, explicit opt-in, and visible disabled/not configured states. |
| 2026-04-25 | Gaming-PC-safe runtime policy is mandatory. | The app targets a gaming PC, not a dedicated server. | Heavy services must be manual, on-demand, mode-based, or disabled; later phases must add stop/health/emergency behavior. |
| 2026-04-25 | Approval, durable jobs, observability, rollback, and emergency stop must precede self-update/full automation. | Self-improving or powerful automations are unsafe without proof, audit, and recovery. | Phase order keeps foundations early; Full Auto is not allowed before safety gates are implemented and verified. |
| 2026-04-25 | Physical actions require simulator/read-only/dry-run/approval/emergency-stop tiers. | Printers, CNC, lasers, shop devices, robots, vacuums, garage systems, and vehicles have real-world risk. | Physical integrations must default safe and record audit events. |
| 2026-04-25 | The current LOCALAI UI identity is protected by the UI Custodian rule. | Existing control center shell and pages are already present. | UI phases may add focused surfaces but must not redesign, reskin, or replace the shell. |
| 2026-04-25 | Phase 00 records current host blockers without treating them as fixed. | AUDIT_REPORT.md shows typechecks/tests/build passed previously, but live API/UI startup, child PowerShell, browser automation, audit, and NVML had environment failures. | Phase 00.5 must repair or explicitly mitigate runtime blockers before feature expansion. |
| 2026-04-25 | Phase 01 runtime controls reuse existing settings, queue, model, audit, route, and Operations systems. | The repo already has SQLite `app_settings`, model unload helpers, async jobs, audit events, thought log, desktop action routes, and an Operations UI. | Do not create a second service registry or UI shell; future services must integrate with `service_policies` and the runtime mode APIs. |
| 2026-04-25 | Emergency Stop does not execute arbitrary stop commands. | The phase requires safe stops, but arbitrary host process kills would violate gaming-PC and physical safety constraints. | Only explicit service-specific safe adapters may auto-stop managed services in later phases; policy stop commands remain metadata until then. |
| 2026-04-25 | Optional cloud/API providers are policy-gated, not routing defaults. | Phase 02 must preserve no-cost local operation and avoid sending local/private data to cloud providers by default. | Ollama remains default; cloud use requires configured provider state, non-blocked data classification, first-use approval, and per-use approval. Provider tests are no-network policy checks in this phase. |
| 2026-04-25 | Approval requests are backed by durable jobs rather than a separate workflow engine. | Phase 03 needed human-in-the-loop state before later automation without adding LangGraph/Temporal or duplicating the existing foundation. | Use `approval_requests` for approval metadata and `durable_jobs` for restart-safe job state/checkpoints/events. Future tools must verify approved payload hashes before execution. |
| 2026-04-25 | Tier 5 and physical P5 actions are denied by software. | Manual-only/prohibited actions should not be queued into executable software paths. | Tier 5/P5 approval requests are recorded and audited as denied/cancelled. Dangerous command execution does not bypass approval with `forceDangerous`. |
| 2026-04-26 | Model lifecycle extends the existing router instead of creating a second model router or provider registry. | The repo already has `modelRolesService`, Ollama gateway tags, provider policy, runtime modes, approvals, audit events, and benchmark rows. | Future model work must reuse `model-lifecycle.ts`, `model-orchestrator.ts`, `role_assignments`, provider policy, and approval queue. Pull/load/unload/delete/replace actions stay approval-gated or dry-run/proposal only. |
| 2026-04-26 | Self-maintainer and repair/update paths are proposal-first. | Self-updating is unsafe without branch isolation, approval, audit, rollback, and test proof. | Updater, system update, repair, model install/unload, and chat-driven self-improvement paths create proposals or approval requests. They must not apply directly to `main`, silently install dependencies, mutate lockfiles, or execute untrusted sources. |
| 2026-04-26 | Tool registry/firewall is the gateway for future MCP/tool execution. | Phase 07A needed one fail-closed policy layer without installing, starting, or executing external tools. Existing plugins, integrations, runtime modes, permissions, approvals, audit, and replay already provide the needed foundation. | Future MCP, Docker MCP Gateway, browser, desktop, integration, and physical tools must register through `tool-registry.ts`, declare explicit scopes, honor runtime/sandbox/approval checks, redact sensitive metadata, and avoid parallel registries or silent execution paths. |
| 2026-04-29 | OpenClaw and NemoClaw are first-class future gateways behind the existing tool firewall, not standalone executors. | Phase 07C needed full future gateway potential without allowing messaging, command, skill install/update, private file, browser/desktop, or physical actions to bypass Phase 07A/07B safety. | Future OpenClaw/NemoClaw work must reuse `claw-gateway.ts`, `tool-registry.ts`, approved profiles, skill quarantine/review state, source allowlists, Docker MCP/future isolation metadata, approval queue, audit/thought logs, and mission replay redaction. |
| 2026-04-29 | Professional RAG extends the existing `lib/rag.ts` hnswlib/SQLite source of truth rather than adding a second RAG engine. | Phase 08A had to preserve personal memory and local hnswlib behavior while adding provider interfaces, source metadata, incremental re-indexing, and citations. | Future evidence/document work must reuse `artifacts/api-server/src/lib/rag.ts`, `routes/rag.ts`, `rag_collections`, `rag_sources`, `rag_chunks`, hnswlib index files, model-orchestrator/Ollama embeddings, and Workspace RAG UI instead of creating duplicate vector stores or ingestion pipelines. Optional parsers/vector stores stay not_configured until explicitly implemented/configured. |
| 2026-05-01 | Automotive diagnostics are a local Master Tech metadata/source-of-truth layer, not a hardware control executor. | Phase 18 needed vehicle profiles, Foxbody project memory, symptom/DTC intake, test-before-parts workflows, repair logs, and provider status without connecting to OBD/CAN/ECU hardware or claiming confirmed faults. | Future vehicle work must reuse `lib/automotive-diagnostics.ts`, Digital Twin refs, Evidence Vault/RAG refs, approval queue, audit/replay redaction, and the `/automotive` UI. OBD/CAN/ECU/scan/write/tune/firmware actions stay not_configured, approval-gated, or manual_only until a later explicit executor phase. |
| 2026-05-01 | Packaging, install, backup, restore, and disaster recovery extend existing rollback/updater/runtime/approval/audit systems. | Phase 21 needed gaming-PC-safe packaging and recovery controls without introducing a second installer, backup engine, or destructive restore path. | Future recovery work must reuse `lib/packaging-recovery.ts`, `routes/system.ts`, existing rollback/snapshot, runtime modes, approval queue, durable jobs, audit/thought logs, and Operations UI. Backup manifests remain metadata-only and local-first; restore execution stays approval-gated and `not_configured` until a later explicit executor with backup, validation, and rollback proof exists. |

## Update rule

Add an entry whenever a phase chooses a major architecture path, rejects an option, changes safety policy, or introduces a new subsystem.
